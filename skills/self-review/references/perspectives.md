# レビュー観点定義

レビュー対象の種類に応じて、対応するセクションの観点を適用する。

## 仕様書

| 観点 | チェック内容 |
|------|-------------|
| 曖昧な記述 | 「適切に」「必要に応じて」等の主観語。実装者が判断に迷う表現 |
| 未定義の振る舞い | エラー時・空データ時・境界値での動作が明記されているか |
| データモデル整合性 | スキーマ定義やDB設計と矛盾がないか |
| 既存仕様との整合性 | 他の仕様書と矛盾・重複がないか |
| 仕様の網羅性 | 操作フロー・バリデーション・表示条件等が明確か |

## 計画書

| 観点 | チェック内容 |
|------|-------------|
| 手順の抜け漏れ | 必要な作業が欠けていないか |
| 依存関係の順序 | 先に作るべきものが後に来ていないか |
| 影響範囲の見落とし | 変更が波及する箇所が網羅されているか |
| 既存パターンとの一貫性 | プロジェクトの規約・アーキテクチャに従っているか |

## コード差分

| 観点 | チェック内容 |
|------|-------------|
| 既存コードとの整合性 | 命名規則・コード構造・パターンが既存と一致しているか |
| 副作用・リグレッション | 変更が意図しない箇所に影響していないか |
| 影響範囲の見落とし | この変更に伴って更新すべきだが漏れているファイルがないか |
| ルール違反 | プロジェクトの規約やルールに違反していないか |

## その他のドキュメント

| 観点 | チェック内容 |
|------|-------------|
| 内容の矛盾 | ドキュメント内で矛盾している記述がないか |
| 既存資産との整合性 | 既存のコード・仕様・ドキュメントと矛盾していないか |
| 情報の欠落 | 読者が判断に必要な情報が不足していないか |
